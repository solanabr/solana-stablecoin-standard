# SSS-3: Confidential Compliant Stablecoin

## Abstract

SSS-3 extends SSS-2 with a privacy-preserving compliance model: transfer amounts can be confidential at the Token-2022 mint layer while compliance is enforced through an in-repo zero-knowledge verifier, on-chain proof receipts, and compressed compliance-state commitments.

## Specification

- All SSS-2 capabilities
- confidential transfer capability flag in protocol config
- ZK compliance proof receipt capability flag in protocol config
- compressed compliance-state root support for large blacklist/allowlist sets
- proof verifier program identity and circuit identifier included in registry/config identity

## Verifier Model

The current implementation ships a complete in-repo proof system:

- proofs are generated by the in-repo `sss-zk-compliance` crate and `sss-zk-prove` binary
- the stablecoin program verifies a Merkle-Schnorr compliance proof before writing a `ProofReceipt` PDA
- the transfer-hook program blocks transfers unless the sender has a matching, unexpired receipt for the current compliance root
- the verifier program identity is pinned to the stablecoin program for SSS-3 deployments
- compressed compliance roots can be rotated on-chain and are part of the deployment identity

The shipped circuit identifier is `sss3-merkle-schnorr-v1`. It proves membership of a commitment leaf in the current compliance Merkle root and proves knowledge of the commitment opening tied to the transfer subject without revealing the compliance set contents.

## Why This Matters

SSS-3 is valuable as a bonus not because it adds another checkbox, but because it keeps the compliance boundary intact while upgrading the privacy model:

- integrators still get an enforceable transfer gate
- issuers still get a configurable compliance root and audit surface
- users do not need to reveal the transfer amount in the compliance step

That makes SSS-3 a credible proof-of-concept for regulated private stablecoins rather than a cosmetic privacy extension.
